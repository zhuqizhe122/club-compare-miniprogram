# 002 — 实现计划

> **Checkpoint 历史说明（2026-07-24）**：本文保留 002 四题确定性评分的实现计划，供回归与迁移对照。专业版目标设计见 `specs/003-adaptive-decision-lab/plan.md`。

## 技术路径

采用原生微信小程序兼容的 CommonJS 模块：

`data/clubs.js`（98 社与查询/比较函数）
→ `utils/match.js`（确定性四题评分）
→ `utils/booth-questions.js`（现场追问）
→ 既有 `pages/list`、`pages/compare`、`pages/result` 消费。

本 slice 不新增页面：`pages/list` 同页承载四题 Top 6 与直接搜索/分类筛选，两种模式共用 2–3 个候选托盘；答案、Top 6 与已选 ID 接入当前会话态。

## 决策

1. 结构化字段由现有原型文案确定性派生，统一标注“待社团确认”。
2. 评分采用可审计的精确枚举匹配，不做模糊画像：分类 4 分、时段 3 分、强度 2 分、互动 2 分。
3. 同分保留原数据顺序，避免随机性和不可复现排序。
4. 推荐结果返回理由与完整计分轨迹，页面无需重新推导解释。
5. 摊位问题依次利用缺失、差异、倾向/偏好和未确认状态，去重后截断为 3 条。
6. Node 测试仅使用内置 `assert`，与小程序业务模块共享同一 CommonJS 实现。
7. 搜索模式不调用评分函数，因此搜索卡不展示匹配理由；只有问卷 Top 6 和带问卷答案的比较页展示可追溯理由。
8. `globalData.preferenceAnswers`、`recommendations` 和 `selectedClubIds` 支撑当前会话内返回恢复；搜索词、分类和模式保留为页面态，不做持久化。

## 页面接入

- `list.onShow()` 从会话态恢复答案、Top 6 和候选，并重新执行当前搜索。
- 四题最后一答调用 `matchClubs()`，生成并缓存 Top 6；改答案或清空答案不会改变已选候选。
- Top 6 与搜索结果都通过同一 `onToggleClub()` 更新选择，最多 3 个；至少 2 个才允许导航到 compare。
- compare 按候选顺序消费选择；result 消费偏好答案以生成现场问题。具体 compare/result 验收归入 `001`。

## 风险控制

- 演示推断风险：数据、卡片、结果均携带待确认状态。
- 规则偏差风险：不使用“最适合”“准确率”等权威措辞，分数只用于排序。
- 兼容风险：不使用 ESM、DOM、Node 专属业务 API 或 npm 依赖。
- 会话风险：只保证当前页面栈与 `globalData` 生命周期，不承诺关闭重开恢复。
- 呈现风险：Top 6、长搜索列表、底部托盘和 3 候选表格需在微信开发者工具中人工验证小屏可读性。
