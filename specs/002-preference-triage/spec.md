# 002 — 社团偏好分流规格

## 目标

新生面对 98 个社团时，可用四个低负担问题得到可解释的 Top 6 候选，再进入既有 2–3 社团比较流程。结果是探索起点，不是“最适合你”的权威结论。

## 用户故事

### US1 — 搜索和结构化筛选

1. 可按名称、简介、分类、标签和别名进行本地子串搜索。
2. 当前选择页 UI 提供分类标签筛选；底层纯函数支持分类、时段、强度、互动方式、门槛和承诺筛选。
3. 搜索和筛选不发起网络请求；空条件在底层返回全部 98 社，页面显示对应结果数量。
4. 搜索无结果时显示明确空态；直接搜索卡只展示社团资料与“待社团确认”，不展示伪造的问卷匹配理由。

### US2 — 四题偏好分流

固定四题依次对应活动分类、方便时段、投入强度和互动方式，权重分别为 4、3、2、2。

验收条件：

1. 相同数据和回答必须得到完全相同的结果。
2. 按总分降序；同分按 98 社原始顺序，最多返回 6 项。
3. 每项提供 2–3 条人类可读理由，并附逐题 expected、actual、matched、points 轨迹。
4. 不使用随机数、网络、第三方依赖或 LLM。
5. 缺答计 0 分，不猜测用户答案。
6. 四题与 Top 6 位于既有 `pages/list`：回答后自动进入下一题，第 4 题后展示 Top 6、每项理由和确认状态。

### US3 — 双模式选择 2–3 个候选

1. `pages/list` 同页提供“帮我缩小范围”和“我已知道名字”两种模式，不新增页面。
2. Top 6 和搜索结果共用同一选择状态与底部托盘；切换模式不得丢失已选候选。
3. 用户必须选择 2–3 个候选后才能进入比较；候选顺序保持点击顺序。
4. 已选 3 个时不得加入第 4 个，并显示明确提示；移除后可以继续选择。
5. 搜索、分类、Top 6、候选托盘都消费同一份本地 98 社数据。

### US4 — 会话态与诚实解释

1. 98 条结构化资料统一为 `dataStatus: 待社团确认`。
2. 分类和匹配是基于已有名称及原型文案的演示级结构化，不冒充官方。
3. 预期卡和现场追问必须继续显示或传递不确定性。
4. 四题答案、Top 6 和已选 ID 保存在 `globalData` 当前会话中，供页面返回时恢复；不写入持久化存储。
5. 搜索词、分类与当前双模式属于选择页页面态；页面销毁或重开后允许重置。
6. “重新开始”清空答案、Top 6、候选和倾向。

## 页面范围与非目标

- 本 slice 覆盖本地数据、纯函数，以及在既有 `pages/list` 内的四题、Top 6、搜索分类、2–3 选择和会话态接入。
- 它与 `001` 共同完成 `list → compare → result` 衔接，但不新增第五个页面。
- 不做人格测评、概率预测、隐式画像、报名或推荐自动决策。
- 不新增 npm 包、后端、网络请求、云开发或持久化。

## 完成定义

- 98 社数量和原 ID 保持不变，ID 与名称唯一。
- 数据符合 `docs/data-contract.md`。
- Node 静态校验和规则测试通过。
- 微信开发者工具人工验证双模式、4 题、Top 6 理由、搜索分类、2–3 边界、顺序和返回状态；未执行前对应任务保持未勾选。
