# 003 — 自适应社团决策实验室规格

> 状态：目标规格，尚未据此宣称实现完成。`001` 与 `002` 是可运行 checkpoint 基线；本规格在保留其“本地优先、可解释、先比较后决定”原则的前提下演进为八页专业版。

## 1. 问题与目标

面对大量社团，新生不仅需要“猜你喜欢”，还需要区分三类信息：不可妥协的硬约束、可权衡的偏好、尚未核实的事实。本功能以 8–14 题自适应评估缩小候选，再通过资料库、社团详情、候选篮、深度比较和结果报告支持可复核决策。

目标不是替用户选定“最优社团”，而是让用户知道：

1. 哪些候选明确违反硬约束，哪些因证据不足只能标记未知；
2. 偏好匹配分数为何是 0–100 中的当前值；
3. 候选在压力情境下是否仍可接受；
4. 哪些结论可行动，哪些必须去现场核实。

## 2. 批准页面与主流程

仅批准以下八个页面，顺序为：

`index → assessment → recommend → library → club → selection → compare → result`

- `index`：说明方法、证据边界、非权威性，并提供“专业测评”“按条件筛选”“直接浏览资料”三个意图入口；后两个入口复用 `library`，不新增页面。
- `assessment`：完成 8 道核心题，并按需要追加最多 6 道自适应题。
- `recommend`：展示经过硬约束判定后的候选和可解释的 0–100 偏好匹配。
- `library`：搜索、筛选和浏览完整本地社团资料。
- `club`：查看单个社团的字段级证据、未知项和加入候选入口。
- `selection`：管理 2–3 个深度比较候选，显示选择理由与证据缺口。
- `compare`：同维对照并执行压力测试。
- `result`：形成非绑定性结果报告和现场核实清单。

允许从 `recommend` 直接查看 `club`，从 `club` 返回来源页；用户仍须经 `selection` 满足 2–3 个候选后进入 `compare`。页面迁移期间旧 `list` 只作为 checkpoint 兼容入口，不属于目标八页。

## 3. 用户故事与验收

### US1（P1）— 低负担完成自适应评估

作为新生，我希望问题数量随我的答案和候选不确定性变化，以免填一份冗长但无用的问卷。

验收条件：

1. 每次会话先问恰好 8 道核心题，覆盖兴趣方向、可用时段、每周投入上限、强制冲突、互动方式、成长目标、参与承诺和技能门槛。
2. 第 8 题后，系统只能因“硬约束仍未知”“前列候选过于接近”或“关键偏好置信不足”追加问题。
3. 总题数不得少于 8、不得多于 14；每道追加题记录触发原因。
4. 下一题由确定性规则从未问题库中选择：优先消除硬约束未知，其次最大化前列候选区分度，再补最低置信偏好维度；同优先级按题目稳定顺序。
5. 若已无可问问题、达到 14 题，或前列候选已经稳定，评估结束；相同数据和相同答案产生相同题序与结果。
6. 用户可返回修改答案；系统从修改点起废弃受影响的后续自适应答案并重新计算，不静默沿用失效结论。
7. 缺答不被猜测；退出评估前必须明确“继续评估”或“放弃本轮”。

### US2（P1）— 先判硬约束，再看偏好匹配

作为用户，我希望绝对不能接受的条件不会被高偏好分数掩盖。

验收条件：

1. 每个候选的每项硬约束结果只能是 `PASS`、`FAIL` 或 `UNKNOWN`。
2. `FAIL` 候选不进入默认推荐区，但可在“被硬约束排除”区查看失败字段与证据。
3. `UNKNOWN` 不等于通过；候选可保留在推荐区，但必须带醒目的“需核实”状态，且不能使用“满足你的硬约束”措辞。
4. `prototype-inferred`（原型推断）证据永远不能产生硬约束 `PASS`；即使推断值表面匹配，也只能得到 `UNKNOWN`。
5. 只有字段值和证据共同满足 `docs/data-contract.md` 的硬通过条件时才可产生 `PASS`。
6. 候选排序先按硬约束状态组（无 `FAIL` 的已核实通过优先于含 `UNKNOWN`），再按偏好匹配、证据置信度和稳定 ID 破同分。

### US3（P1）— 理解 0–100 多准则偏好匹配

作为用户，我希望看到一个可解释的匹配度，而不是人格标签或黑箱推荐。

验收条件：

1. 匹配覆盖兴趣、时段偏好、投入偏好、互动方式、成长目标、参与方式六个软偏好维度。
2. 用户权重归一化后总和为 100；每维兼容度为 0–100，最终偏好匹配为各维兼容度的加权平均并四舍五入为整数 0–100。
3. 公式不把证据等级伪装成喜好：偏好匹配与证据置信度分别显示。
4. 字段未知时该维兼容度使用中性值 50，同时降低独立的证据置信度；报告必须列出未知维度，禁止把 50 解释为“中等适合”。
5. 每个推荐展示总分、最多 3 个主要加分因素、主要冲突、硬约束状态和证据置信度；可展开查看逐维权重、兼容度、贡献和证据。
6. 修改答案后所有分数、排序、理由和选择快照必须重新计算或标记过期。
7. 不输出心理类型、成功概率、“最适合你”或准确率承诺。

### US4（P2）— 从推荐或资料库形成候选篮

作为用户，我希望既能接受系统缩小范围，也能主动查找已知社团。

验收条件：

1. `recommend` 和 `library` 使用同一份本地数据、相同字段级证据和同一候选篮。
2. 首页“按条件筛选”直接进入展开筛选器的 `library`；“直接浏览资料”进入同一资料库的默认浏览态，二者不得形成两套结果或两套候选状态。
3. 资料库支持名称/别名/标签搜索，以及分类、时段、投入、互动、活动节奏、活动形式、门槛和证据状态筛选；不同字段之间取交集，无结果时显示可恢复的空态。
4. 每个筛选结果均可进入现有 `club` 详情；从详情返回后，筛选条件、结果页位置和候选篮保持不变。
5. `club` 对每个决策字段显示值、A–D/U 证据等级、来源类型、更新时间或“未知”，不得只显示对象级笼统标签。
6. 用户可选择 2–3 个候选；少于 2 个不能比较，达到 3 个不能加入第 4 个，移除后可继续。
7. 候选篮记录来源（推荐、资料库或详情）、加入时匹配快照和当前是否已过期；不得把搜索命中伪装成推荐理由。

### US5（P2）— 深度比较并进行压力测试

作为用户，我希望比较候选在现实压力下的表现，而不只看常态宣传。

验收条件：

1. 比较页按候选加入顺序展示 2–3 列，并对齐硬约束、六个软偏好维度、现有四项基础比较信息及字段证据。
2. 差异高亮只依据规范化值；未知值明确显示“未知/待核实”，不得推断补齐。
3. 压力测试至少包含“考试周时间减半”“连续两次无法参加”“任务临时加码”三个标准情境，并可选择一个与用户答案相关的情境。
4. 每个“候选 × 情境”结果只能是 `RESILIENT`（仍可接受）、`AT_RISK`（有风险）或 `UNKNOWN`（证据不足），并展示触发规则和所用字段。
5. 压力测试不改变原始社团资料，不生成概率，不把 `prototype-inferred` 当成可靠承诺；证据不足必须输出 `UNKNOWN`。
6. 用户可返回调整候选或评估答案；过期比较结果不得继续用于最终报告。

### US6（P1）— 获得可复核的结果报告

作为用户，我希望带走一份包含结论边界和现场问题的报告。

验收条件：

1. 用户可选择“当前更倾向某候选”“保留两个候选继续核实”或“暂不选择”。
2. 报告包含会话摘要、候选排序、选择理由、硬约束三值结果、0–100 偏好匹配、证据置信度、压力测试摘要、未知项和现场核实问题。
3. 报告中的事实均可追溯到字段级 evidence；推断内容必须逐项标识，不能只在页尾统一免责。
4. 现场问题优先覆盖硬约束 `UNKNOWN`、压力测试 `UNKNOWN/AT_RISK` 和低等级关键证据，去重后最多 5 条。
5. 报告是当前会话内的非绑定性决策辅助，不声称保存、同步、官方认证或已经报名；真实加入仍在线下进行。
6. “重新开始”清除八页会话状态并回到首页。

## 4. 关键业务规则

### 4.1 自适应停止规则

完成 8 道核心题后，在下列任一条件成立时停止：已达 14 题；没有合法未问题；所有硬约束都有可判定答案，且推荐前 3 名相邻分差均至少 8 分，且六个软维度置信均达到既定阈值。否则按 US1 的优先级选择下一题。

“8 分”是原型可验收阈值，不代表统计显著性；后续变更必须先更新规格和决策 ADR。

### 4.2 偏好匹配

对六个维度分别得到用户权重和候选兼容度。归一化权重之和为 100，最终匹配为：

**偏好匹配 = 四舍五入〔各维（归一化权重 × 兼容度）的总和 ÷ 100〕**

硬约束不进入该公式。证据等级也不直接乘入分数，而是产生独立的 0–100 证据置信度，避免“资料较全”被误解为“更喜欢”。

### 4.3 证据等级

- A：当前有效的官方明确资料，可定位到具体来源。
- B：经社团负责人或至少两条独立可靠来源交叉确认。
- C：单一访谈、成员经验或有时间边界的二手资料。
- D：仅有弱线索或未经交叉确认的陈述。
- U：未知、缺失、来源不可定位或仅为原型占位。

完整来源类型与硬约束判定资格以 `docs/data-contract.md` 为准。

## 5. 八页会话状态

会话至少包含以下分区，均只存在当前运行会话：

1. `navigation`：当前页、来源页、会话版本。
2. `assessment`：题目版本、已问题 ID、答案、触发原因、完成状态。
3. `decisionProfile`：硬约束、六维偏好、原始权重和归一化权重。
4. `recommendation`：计算版本、候选结果、排除结果、证据置信度。
5. `library`：搜索词、筛选器、排序方式。
6. `clubView`：当前社团 ID、进入来源。
7. `selection`：2–3 个候选 ID、来源、匹配快照、过期标记。
8. `comparison`：候选快照、差异、压力情境与结果、计算版本。
9. `report`：当前结论、报告快照、核实问题、生成版本。

状态转换要求：

- 修改任何评估答案都会递增计算版本，并使推荐、选择快照、比较和报告失效。
- 删除候选会使比较和报告失效；排序或浏览筛选不使决策结果失效。
- 页面进入时必须校验前置状态；缺失时回到最近可恢复页面，而不是展示空白或使用陈旧数据。
- 不写入 Storage，不承诺关闭重开、跨设备或跨会话恢复。

## 6. 边界与异常

- 若全部候选均 `FAIL`，推荐页显示排除原因并引导修改硬约束或浏览资料库，不偷偷放宽条件。
- 若全部非失败候选均含 `UNKNOWN`，明确显示“当前没有可核实为通过的候选”。
- 若权重总和为 0，使用六维等权默认值并在报告中标注。
- 若字段值与 evidence 缺一，按未知处理并记录数据错误，不降级为可用事实。
- 若题库无法继续区分候选，停止追加并说明“现有资料不足以进一步区分”。
- 若候选数据在会话中发生版本变化，旧快照失效，用户须重新确认候选。

## 7. Non-goals

- 不做人格测评、心理诊断、职业测评或“MBTI 式”类型标签。
- 不做 AI/LLM 推荐、概率预测、协同过滤、用户画像训练或自动替用户决定。
- 不做登录、账号、云同步、远程数据库、后端 API、跨会话保存或多人协作。
- 不做小程序内扫码、加群、报名、支付、负责人联系或官方认证。
- 不采集分析埋点，不引入新 npm 依赖，不增加八页之外的产品页面。
- 不把原型推断、演示数据、匹配分或压力测试写成社团承诺。

## 8. 可衡量成功标准

1. 测试者可在 8–14 题内完成评估；任何路径都不会出现第 15 题。
2. 相同输入重复运行 20 次，题序、三值判定、分数、排序和理由完全一致。
3. 所有 `prototype-inferred` 硬约束样例均为 `UNKNOWN`，零例产生 `PASS`。
4. 每个 0–100 分数可由逐维贡献重新计算，重算误差为 0。
5. 用户可在 3 分钟内从推荐或资料库组成 2–3 个候选并进入比较。
6. 每份结果报告都包含至少一个证据边界说明；存在未知时，报告完整列出而不隐藏。
7. 八页主路径、返回修改、全失败、全未知、2/3 候选和重开边界均有可重复验收步骤。

## 9. 完成定义

- 规格、计划、任务、数据契约与 ADR 一致，且无未决占位符。
- 目标八页和状态迁移按 `quickstart.md` 在微信开发者工具中通过人工验收。
- 规则、数据、路由和页面状态自动测试通过；自动测试不替代人工 UI 证据。
- 未执行的实现或人工验收任务保持未勾选，不得因本文档存在而宣称专业版已完成。
