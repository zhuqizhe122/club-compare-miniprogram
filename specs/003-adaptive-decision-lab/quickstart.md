# 003 — 八页专业版快速验证与验收指南

> 当前代码已经形成八页实现，但“已实现”不等于微信 CLI、微信开发者工具或真机已经验收。所有人工步骤在实际执行并留证前均视为未通过。

## 1. 前置条件

1. 从仓库根目录导入微信开发者工具，可使用测试 AppID。
2. 无需安装 npm 包、启动服务、登录或联网。
3. 核对 `app.json` 精确注册：
   `index、assessment、recommend、library、club、selection、compare、result`。
4. 当前 98 社运行数据统一为 D 级 `prototype-inferred`，适合验证 UNKNOWN 和证据降级；PASS/FAIL 只能使用测试中的 A 级合成夹具验证，不得把生产原型数据口头升级成已核实资料。

## 2. 自动验证

在仓库根目录执行：

```text
node scripts/validate-project.js
node scripts/validate-data.js
node scripts/validate-engine.js
node tests/decision-rules.test.js
node tests/professional-engine.test.js
node tests/page-state.test.js
```

预期所有命令退出码为 0。重点断言：

- 核心题恰好 8 道，总题数始终在 8–14；
- 相同输入的题序、三值、分数、排序和解释确定不变；
- `prototype-inferred` 永远不能产生硬约束 PASS；
- 逐维贡献可精确重算整数 0–100 偏好匹配；
- 未知字段使用中性计算值但降低独立置信度；
- 候选或答案变化会使比较和报告失效；
- `app.json` 当前仅注册批准八页。

逐条记录命令、日期和退出码；不得用“CLI 已通过”概括 Node 输出。Node 只验证脚本中的静态或纯函数断言。

## 3. 主路径：最短 8 题

1. 首页确认可见“决策辅助、8–14 题、资料证据有等级、真实加入在线下”，开始新会话。
2. 完成 8 道核心题，并在每题核对题号、单选/多选类型和“为什么问”。
3. 只有在停止条件实际满足时，确认第 8 题后直接进入推荐页；若当前 D 级候选仍触发追加题，应按页面继续，不得为了演示声称必然 8 题结束。
4. 推荐页确认：
   - Top 6 显示稳定名次、0–100 偏好匹配分和可读理由；
   - 现实代价与硬条件未知分开显示；
   - UNKNOWN 明确写“待核实”，不声称满足硬约束；
   - 偏好匹配分与资料等级 D 明确分离。
5. 对照自动测试的解释轨迹核对 0–100 分可由逐维贡献重算；当前 UI 不提供完整逐维展开时，不把该项记作页面验收通过。
6. 从推荐进入详情，确认字段值、D 级来源类型和“不能作为硬条件通过依据”的说明可见。
7. 加入一个推荐候选；转到资料库搜索另两个候选并加入，确认候选篮来源不同但状态共享。
8. 在候选页依次保留 3 个候选，确认顺序不变；尝试第 4 个被阻止。
9. 进入比较，确认候选摘要、关键 3 个权衡、分组字段值、证据和未知项对齐；依次运行 `busy`、`deep`、`social` 三个压力情境。
10. 对每个压力情境确认基础分、情境分、分差和名次变化真实刷新，并显示“D 级演示估计，非预测”。
11. 选择“当前更倾向某候选”，生成报告；确认只生成一张主倾向卡，包含匹配分、D 级资料、理由、代价、未知和 3–5 条现场问题。
12. 再验证“先都不加”能生成暂缓行动卡。
13. 确认报告没有“最适合”“成功概率”“官方认证”或已报名暗示。

## 4. 自适应与回退路径

### 最长 14 题

使用能保持硬约束未知、前列分数接近或低置信维度的夹具：

1. 第 8 题后确认继续追加，并逐题显示追加原因。
2. 追加原因应对应：硬约束未知、头部候选区分或偏好覆盖不足。
3. 第 14 题后必须结束；不得出现第 15 题。
4. 若资料不足以区分，页面明确说明，不假造新结论。

### 修改答案

1. 从推荐返回评估，修改影响硬约束或权重的早期答案。
2. 确认受影响的后续自适应答案被废弃并重新规划。
3. 确认旧推荐、候选、压力情境和倾向被清空，并按新答案重新生成。
4. 未重新确认前，不得使用旧比较或旧报告。

## 5. 硬约束与证据边界

分别验证：

1. 合格 A/B 证据且明确满足 → PASS。
2. 合格证据且明确冲突 → FAIL，并展示具体字段和来源。
3. 值缺失、evidence 缺失或等级 U → UNKNOWN。
4. 值表面匹配但来源为 `prototype-inferred` → UNKNOWN，绝不 PASS。
5. 当前生产数据应主要呈现 UNKNOWN 与 D 级降级，不得把 D 级资料显示成 PASS。
6. 全 FAIL、全 UNKNOWN 的分区/空态若无可操作的 UI 夹具，只记录底层自动断言，不勾选页面人工项。

## 6. 压力测试

对 2 个和 3 个候选分别执行三个当前已接入情境：

- 考试周时间减半；
- 连续两次无法参加；
- 任务临时加码；
确认页面展示基础分、情境分、分差和名次变化；三种结果都明确是 D 级演示估计而非预测。底层 `RESILIENT / AT_RISK / UNKNOWN` 证据门禁由 Node 测试验证，不应与页面的情境重排分数混为一谈。

## 7. 状态、返回与断网

1. 从 `recommend/library/club` 多次往返，候选篮保持一致。
2. 删除候选后，旧比较和报告立即失效。
3. 少于 2 个不能比较，2–3 个可以，第 4 个被拒绝。
4. 缺少有效 2–3 候选直接进入比较或结果页时，显示守卫信息，不生成伪报告。
5. 点击重新开始后，八页会话分区全部清空并回首页。
6. 关闭重开不要求恢复，界面不得声称已保存。
7. 断网走完整主路径，Network 面板无业务请求。
