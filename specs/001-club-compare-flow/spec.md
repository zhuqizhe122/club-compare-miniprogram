# 001 — 社团对照主流程规格

> 本规格描述当前四页决策罗盘 UI 基线；偏好分流与搜索详见 `specs/002-preference-triage/spec.md`。

## 用户与目标

目标用户是准备了解或加入大学社团的新生。用户可以把 2–3 个候选社团放入同一张对照表，再记录一个非绑定性的倾向。

## 范围

流程固定为：

`pages/index` → `pages/list` → `pages/compare` → `pages/result`

### US1 — 完成比较并记录倾向

作为新生，我希望比较候选社团并记录倾向，以便在线下加入前做出更清楚的判断。

验收条件：

1. 首页说明“加入前先比一比”的用途，并提供开始入口。
2. 选择页展示至少 3 个本地样例社团的名称和简介。
3. 比较页并排展示所选社团的每周时间、活动频率、普通成员职责和氛围标签。
4. 结果页允许选择一个已选社团或“先都不加”。
5. 确认后生成行动卡，并提示真实加入在线下渠道完成。

### US2 — 限制比较数量

作为用户，我希望系统明确限制候选数量，以免对照表难以阅读。

验收条件：

1. 少于 2 个社团时不能进入比较页，并显示明确提示。
2. 最多选择 3 个社团；选满后未选项不可继续选择，并显示说明。
3. 比较页和结果页只展示当前选中的社团。

### US3 — 诚实展示缺失信息

作为用户，我希望缺失信息被明确标识，以免样例数据被误认为真实完整资料。

验收条件：

1. 四个比较字段为 `null`、`undefined`、空字符串或仅空白时，界面显示“未提供”。
2. 不根据其他字段推断或编造缺失信息。

### US4 — 看懂差异并准备现场确认

作为用户，我希望对比结果指出真正不同的维度，并把不确定信息转成现场问题。

验收条件：

1. 比较页按用户选择顺序展示候选；值不同的四维行必须高亮。
2. 关键差异摘要只包含候选值不完全相同的维度，并保留每个值对应的社团；无差异时给出诚实提示。
3. 用户确认某社团倾向后，结果页生成该社团的中文预期卡，概括投入、互动、门槛和时段。
4. 用户选择“先都不加”后，结果页生成对应的中文行动卡，不暗示必须加入。
5. 社团预期卡明确显示“待社团确认”，不得暴露内部英文枚举或把演示级推测写成官方资料。
6. 结果页根据资料缺失、候选差异、当前倾向/偏好和未确认状态生成最多 3 条摊位追问；当前完整数据主路径应展示 3 条。
7. 比较页提供返回调整；结果页提供返回调整和重新开始。

## 数据与状态

- 数据对象和辅助函数以 `docs/data-contract.md` 为准。
- 社团数据只来自 `data/clubs.js`。
- 已选 ID、四题答案、Top 6 和倾向只保存在当前会话的 `globalData` 中；搜索词、分类和当前模式属于选择页页面态。
- 本 slice 不承诺跨启动持久化。

## 不在本规格内

登录、后端、远程数据、AI/LLM 推荐、扫码、加群、报名、支付、分析 SDK 和新页面均不在本规格内。四题偏好分流、Top 6、搜索与分类筛选由 `specs/002-preference-triage/` 定义并在既有选择页内实现。

## 完成定义

- 静态检查无阻断问题。
- Node 静态校验与规则测试通过。
- 按 `quickstart.md` 在微信开发者工具中完成人工主路径和边界回归；未实际执行的人工项必须保持未勾选。
- 行为变化同步更新规格与 `validation-checklist.md`；回归修复同步更新 `fix-log.md`。
