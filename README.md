# 社团决策实验室

面向第一次接触大量社团的新生。本分支已经接入本地优先的八页专业决策流程：先用 8–14 题区分硬约束与偏好，再以确定性 MCDA（多准则决策分析）形成 Top 6，随后从推荐或资料库保留 2–3 个候选，完成深度比较、压力情境重排和非绑定性报告。

它不替用户宣布“最优社团”，而是把匹配、现实代价、未知项和资料可靠性分开呈现，帮助用户知道该比较什么、还要去现场问什么。

## 已实现的八页流程

1. `pages/index`：说明专业决策方法，并提供专业测评、按条件筛选、直接浏览资料三个入口和 D 级证据边界；后两个入口复用资料库。
2. `pages/assessment`：8 道核心题加最多 6 道自适应题；每题显示“为什么问”，最少 8 题、最多 14 题。
3. `pages/recommend`：展示 Top 6、0–100 偏好匹配分、理由、现实代价、硬条件未知项，并把匹配分与资料等级分离。
4. `pages/library`：搜索 98 个本地社团，并按分类、时段、投入、互动、活动节奏、活动形式、入门门槛和证据状态组合筛选；结果可进入现有详情页。
5. `pages/club`：展示单个社团的时间、参与方式、门槛、成本、五维体验及 D 级来源边界。
6. `pages/selection`：按加入顺序管理共享的 2–3 个候选；少于 2 个不能比较，第 4 个会被拒绝。
7. `pages/compare`：展示候选摘要、关键 3 个权衡、分组字段比较，以及 `busy`、`deep`、`social` 三种压力情境下的分数与名次重排。
8. `pages/result`：让用户选择一个主倾向或“先都不加”，生成一张非绑定性行动卡、3–5 个现场核实问题和四周复盘标准。

`app.json` 当前只注册以上八页。旧 `pages/list` 文件仍作为历史 checkpoint 留在仓库，但不在当前路由中。

## 决策方法

- **自适应提问**：前 8 题建立基本画像；若硬条件仍未知、头部候选接近或偏好覆盖不足，再确定性追加问题，总数不会超过 14。
- **硬约束门禁**：规则支持 `PASS / FAIL / UNKNOWN` 三值判断。当前资料均为 D 级 `prototype-inferred`，因此不能仅凭表面匹配给出硬约束 PASS。
- **MCDA 偏好评分**：把多个偏好维度按权重与兼容度汇总为 0–100 分；无偏好维度不进入分母。分数表示“与当前答案的匹配程度”，不是社团质量排名。
- **分数与证据分离**：高匹配分不代表资料可靠。当前 98 社的决策字段均带 D 级原型推断证据，必须在线下逐项核实。
- **压力重排**：比较页可切换“考试周时间减半”“任务临时加码”“连续两次无法参加”，观察情境分与候选名次如何变化。结果是 D 级演示估计，不是预测。
- **非绑定性报告**：报告只记录当前主倾向、需要接受的代价、未知项和核实问题；不代表报名、官方认证或最终承诺。

## 数据与系统边界

- 98 个社团、问题、规则和评分均保存在本地；代码静态边界禁止业务网络请求、登录、云开发、持久化存储和扫码。
- 不使用 AI、LLM、随机数或第三方推荐服务；相同输入应得到可复现的题序、评分和排序。
- 社团名称来自当前名单；画像、投入、费用、选拔和活动体验主要是原型推断，不可当作本学期官方信息。
- 答案、Top 6、候选、压力情境和倾向只保留在当前运行会话；关闭重开后不承诺恢复。
- 小程序不完成扫码、加群或报名；真实加入仍通过招新摊位、官方群聊或报名渠道进行。

## 导入微信开发者工具

1. 打开微信开发者工具，选择“导入项目”。
2. 目录选择本仓库根目录，即同时包含 `project.config.json` 与 `app.json` 的目录。
3. AppID 可使用测试号；无需安装 npm 依赖，也无需启动服务器。
4. 按 `specs/003-adaptive-decision-lab/quickstart.md` 和 `validation-checklist.md` 执行人工验收。

本文不声明微信 CLI 或微信开发者工具已经验收通过；页面布局、交互、返回路径、断网行为和真机表现仍须人工执行后留证。

在仓库根目录逐条执行：

```text
node scripts/validate-project.js
node scripts/validate-data.js
node scripts/validate-engine.js

node tests/professional-engine.test.js
node tests/page-state.test.js
```
