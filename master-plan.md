# 项目主计划 — club-compare-miniprogram

> 来源说明：本文档于 2026-07-21 根据仓库现有代码、`README.md`、`AGENTS.md`、
> `validation-checklist.md` 和 `fix-log.md` 整理，用于建立最小协作基线；它不是对缺失课程文档的原文恢复。

## 产品问题

大学新生在加入社团前通常只能看到宣传口号，难以快速了解真实活动、时间投入、普通成员职责和社团氛围。

## 产品目标

提供一个本地优先的微信小程序决策原型，让用户通过 8–14 题自适应评估区分硬约束与软偏好，在推荐和资料库中核查字段级证据，选择 2–3 个候选进行深度比较和压力测试，最后形成可复核但不绑定的报告：

- 当前更倾向某个候选；
- 保留两个候选继续核实；或
- 暂不选择。

真实扫码、加群、联系负责人和报名均在线下渠道完成。

## Checkpoint 基线与目标

已存在的 001/002 checkpoint 是四页 walking skeleton：

`pages/index` → `pages/list` → `pages/compare` → `pages/result`

它用于回归当前四题、Top 6、搜索、2–3 候选、基础比较和倾向流程，不因专业版计划而改写历史。

003 的批准目标为八页：

`pages/index` → `pages/assessment` → `pages/recommend` → `pages/library` → `pages/club` → `pages/selection` → `pages/compare` → `pages/result`

专业版尚须按 `specs/003-adaptive-decision-lab/tasks.md` 实施和验收；本文只批准方向，不宣称八页已经完成。

## 专业版核心能力

1. **8–14 题自适应评估**：8 道核心题后，只为硬约束未知、前列候选接近或低置信偏好追加，最多 14 题。
2. **硬约束三值门禁**：每项只为 PASS、FAIL、UNKNOWN；UNKNOWN 不等于通过。
3. **多准则偏好匹配**：六维权重归一化为 100，输出可重算的 0–100 偏好分；证据置信度另行显示。
4. **字段级 evidence**：每个决策事实标 A–D/U、来源和时间；`prototype-inferred` 永远不能支持硬 PASS。
5. **双路径探索**：推荐和资料库共用详情与 2–3 候选篮。
6. **深度比较压力测试**：至少覆盖考试周、连续缺席和任务加码，输出 RESILIENT、AT_RISK、UNKNOWN。
7. **结果报告**：聚合结论、三值、偏好分、置信度、压力结果、未知项和最多 5 条现场核实问题。

## 非目标

未经单独批准，不增加：

- 登录、云同步、微信云开发或远程数据库；
- 后端 API、实时 AI/LLM 推荐；
- 支付、数据分析 SDK；
- 小程序内扫码、加群或报名；
- 新 npm 依赖和无关模板页面。
- 人格测评、心理类型、概率预测、用户画像训练或“最适合你”的权威结论；
- 将原型推断、匹配分或压力测试包装为社团官方承诺；
- 批准八页之外的产品页面。

## 专业版里程碑

1. **M0 — checkpoint 冻结**：保留 001/002 自动与人工回归语义。
2. **M1 — 合同和纯函数**：字段级 evidence、题库、会话状态、三值门禁、匹配、压力和报告规则；状态为待实施。
3. **M2 — 评估与推荐**：`index → assessment → recommend` 可独立演示 8–14 题和可解释推荐；状态为待实施。
4. **M3 — 探索与候选**：`library → club → selection` 共用资料与候选篮；状态为待实施。
5. **M4 — 比较与报告**：`compare → result` 完成压力测试和可追溯报告；状态为待实施。
6. **M5 — 八页收口**：精确路由、自动测试、DevTools 主路径/边界证据和 checkpoint 回归全部通过后才可声明专业版完成。

## 协作门槛

- 不直接在 `main` 开发；一个 Issue 对应一个负责人、一个分支和一个 PR。
- 共享数据结构、`app.json`、`app.js` 和 `data/clubs.js` 由单一负责人顺序修改。
- 行为变化先核对规格；数据结构变化先更新数据契约。
- 自适应停止阈值、硬 PASS 资格、证据等级、评分公式、压力规则或路由变化必须同步 ADR。
- PR 前运行静态检查，并由人工在微信开发者工具中走查相关流程。
- 人工查看 `git diff` 后才允许 commit；PR 由另一名成员 Review 后合并。

## 验收边界与未决问题

- 结构化演示字段仍需社团逐条确认后，才能升级为已核实资料。
- 当前仅有代码走查与 Node 规则证据；双模式、3 列可读性、返回/重开、断网等必须在微信开发者工具中人工验收。
- 当前会话态不持久化，关闭或重开后不承诺恢复；这是明确边界，不是数据丢失缺陷。
- GitHub 分支保护和 Review 强制规则尚未确认。
- 98 社字段级证据尚未审计；迁移时大量专业字段保持 UNKNOWN 是正确行为，不得批量伪造 A/B。
- 八页专业版的布局、返回路径、最短/最长题序、三列比较和报告理解度尚无人工证据。
