# 项目主计划 — club-compare-miniprogram

> 来源说明：本文档于 2026-07-21 根据仓库现有代码、`README.md`、`AGENTS.md`、
> `validation-checklist.md` 和 `fix-log.md` 整理，用于建立最小协作基线；它不是对缺失课程文档的原文恢复。

## 产品问题

大学新生在加入社团前通常只能看到宣传口号，难以快速了解真实活动、时间投入、普通成员职责和社团氛围。

## 产品目标

提供一个本地优先的微信小程序原型，让用户先通过四题偏好分流缩小范围，再选择 2–3 个候选社团，并排比较固定维度、查看差异与预期，最后记录：

- 更倾向某个已选社团；或
- 先都不加。

真实扫码、加群、联系负责人和报名均在线下渠道完成。

## 当前交付基线

当前 walking skeleton 已包含四页流程：

`pages/index` → `pages/list` → `pages/compare` → `pages/result`

数据来自 `data/clubs.js`。选择页已同页提供四题 Top 6 和直接搜索两种模式；比较页已接入差异摘要与行高亮；结果页已接入倾向、“先都不加”、中文预期卡和最多三条现场追问。答题、Top 6、选择和倾向仅保存在当前会话的 `globalData` 中。正式验收标准见 `specs/001-club-compare-flow/` 与 `specs/002-preference-triage/`，数据字段见 `docs/data-contract.md`。

## 非目标

未经单独批准，不增加：

- 登录、云同步、微信云开发或远程数据库；
- 后端 API、实时 AI/LLM 推荐；
- 支付、数据分析 SDK；
- 小程序内扫码、加群或报名；
- 新 npm 依赖和无关模板页面。

## 阶段

1. **可运行原型**：四页主流程和本地样例数据，已完成。
2. **本地数据扩展**：保留 98 个社团并补充演示级分类、标签、投入和互动字段；所有数据明确“待社团确认”，已完成。
3. **偏好分流**：四题确定性加权评分给出 Top 6，理由可追溯；同页搜索和分类筛选；已接入选择页。
4. **候选决策支持**：2–3 个候选按选择顺序比较，差异高亮；倾向或“先都不加”后生成中文预期卡与最多三条摊位追问；已接入比较页和结果页。
5. **验收交付**：Node 校验、微信开发者工具人工清单、2–3 分钟演示脚本和回归风险记录。自动校验完成，DevTools 人工项待执行。

## 协作门槛

- 不直接在 `main` 开发；一个 Issue 对应一个负责人、一个分支和一个 PR。
- 共享数据结构、`app.json`、`app.js` 和 `data/clubs.js` 由单一负责人顺序修改。
- 行为变化先核对规格；数据结构变化先更新数据契约。
- PR 前运行静态检查，并由人工在微信开发者工具中走查相关流程。
- 人工查看 `git diff` 后才允许 commit；PR 由另一名成员 Review 后合并。

## 验收边界与未决问题

- 结构化演示字段仍需社团逐条确认后，才能升级为已核实资料。
- 当前仅有代码走查与 Node 规则证据；双模式、3 列可读性、返回/重开、断网等必须在微信开发者工具中人工验收。
- 当前会话态不持久化，关闭或重开后不承诺恢复；这是明确边界，不是数据丢失缺陷。
- GitHub 分支保护和 Review 强制规则尚未确认。
